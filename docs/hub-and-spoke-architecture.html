Hub-and-Spoke Architecture

Multi-source data architecture for Czech public registers, unifying company data through a shared hub model keyed by ICO — with denormalized search fields for cross-source multi-parameter filtering.

Approved 2026-02-27

1 The Problem: Isolated Data Silos

ARES

Real-time API fetches cached only in Redis (volatile)

No shared entity links the two sources. As more Czech public registers are added (Trade Register, ISIR, RES, ISVZ…), each new source would create another isolated silo without a unifying structure.

3 Why This Approach?

Unified Canonical Model

A "fat" Company with all fields from all sources merged into one.

Over-engineered

Complex merge logic from day 1. Multi-parameter search only needs ~6 fields, not ~100+.

Hub-and-Spoke + Every-Touch-Persists

Lightweight Company hub (ICO key) with denormalized search fields. Each source stores its own models as spokes via FK. Every API interaction persists to DB (3-tier: Redis → DB → API).

Just right

DB grows organically from usage. Stale records served instantly with background refresh. Reduces rate-limited API calls over time.

No Hub / Query-Time Join

No central entity. Join across sources on-the-fly at query time.

N+1 complexity

Cross-source filtering (ARES region + Justice revenue) requires in-Python intersection — impossible to paginate.

5 Service Layer

▶ JusticeSyncService._upsert_entity() — Bulk import flow
Parse ICO from XML subjekt node
Company.objects.get_or_create(ico=ico) — find or create hub record
Set name and is_active from subjekt data
Phase 2: Set legal_form on Company if not already set by ARES
Entity.objects.create(company=company, ...) — link spoke
▶ AresService.get_by_ico() — 3-tier lookup: Redis → DB → API
L1: Redis — check hot cache (existing behavior)
L2: DB — EconomicSubject.objects.filter(ico=normalized) — if found with raw_data, parse and return immediately
If L2 hit but stale (>24h): trigger _schedule_background_refresh() via daemon thread (non-blocking)
L3: ARES API — rate-limited fetch, parse, persist via _persist_detail()
_persist_detail(): Company.objects.update_or_create() with search fields + EconomicSubject.objects.update_or_create()
Phase 2: Populates legal_form, region_code, employee_category, nace_primary
Redis cache always updated on any fetch (L2 or L3)
▶ AresService.search() — Every-touch-persists
Existing: check Redis cache → throttle → call ARES search API → parse
NEW: _persist_search_results() — for each entity in results:
  Company.objects.get_or_create(ico=...) — creates hub record (does NOT overwrite detail data)
  EconomicSubject.objects.update_or_create(ico=...) — saves summary raw_data
Redis cache for individual subjects (existing behavior preserved)
DB grows organically from every search — enriches local search for sales team
▶ CompanyService.get_by_ico() — Unified lookup (new)
Company.objects.get(ico=ico.zfill(8)) — find hub
Collect company.ares_records.first()
Collect company.justice_entities.select_related().prefetch_related("facts").first()
Return unified dict: { "company", "ares", "justice" }
▶ CompanyService.search() — Multi-parameter search (Phase 2)
Start with Company.objects.all()
Apply filters: legal_form, region_code, employee_category, revenue range, NACE, status
Single-table query — all filters on indexed Company columns, no JOINs
Paginate with offset / limit, order by -latest_revenue
Return list of CompanySummary dicts with search field values

8 Adding Future Sources

Repeatable pattern for any new Czech public register:

Create a new Django app (e.g. trade_register)
Define source-specific model with company = ForeignKey("company.Company")
Implement fetcher — bulk sync or real-time API, depending on source
In fetch/sync code: Company.objects.get_or_create(ico=...) then link
If the source provides new searchable data (e.g. ISIR insolvency status), add a denormalized field to Company and populate during fetch/sync
Add source data to CompanyService and CompanyDetailView
✓

No changes to existing spokes needed. Each new source is purely additive — new search fields on Company are additive migrations, and the hub-and-spoke pattern ensures total isolation between sources.
